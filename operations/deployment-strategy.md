# Deployment Strategy

Last updated: 2026-04-19.

## Intent

This document defines how AxiomNode promotes runtime changes across environments, how image versioning is used, and what is considered automatic versus controlled release behavior.

## Environment model

- `dev`: developer-owned local runtime, primarily Docker Compose based, optimized for fast iteration and low coordination cost
- `stg`: shared remote staging on the VPS k3s cluster, used as the automatic deployment target for covered services
- `prod`: controlled production target with manual promotion, stricter operational review, and higher runtime guarantees

## Deployment principles

- service repositories validate service correctness
- `platform-infra` owns packaging and deployment policy
- automatic deployment is allowed only where rollback cost and operational blast radius are acceptable
- production promotion must be explicit, inspectable, and reversible
- mutable image tags may exist for operator convenience, but automated staging rollout must be pinned to immutable artifacts

## Current strategy by environment

| Environment | Trigger model | Image selection model | Restart model | Control level |
|---|---|---|---|---|
| `dev` | local/manual | local containers or `dev` tags | operator-driven | low ceremony |
| `stg` | automatic from covered services on `main` | immutable short-SHA from triggering run | apply-based rollout without forced restart for immutable tags | medium |
| `prod` | manual only | mutable `prod` tag or explicitly selected release artifact | controlled rollout | high |

## Automatic staging deployment

Automatic deployment to staging is enabled for the covered runtime services.

Conditions:

1. a change is pushed to `main` in a covered service repository
2. the service validation jobs succeed
3. the service workflow dispatches `platform-infra` build-push
4. the centralized image build succeeds
5. `platform-infra` deploy runs via `workflow_run`

Staging deployment therefore acts as a continuation of CI, not as a manually curated release lane.

## Production promotion

Production is intentionally not part of the automatic path.

Reasons:

- production may require coordinated schema, secret, ingress, and operational readiness checks
- production changes often need business-aware rollout timing
- production rollback decisions should remain explicit and operator-owned

Promotion to production should continue to use a manual dispatch in `platform-infra`, optionally coupled to a release checklist and operational approval.

## Image versioning strategy

### Immutable tags

The short Git SHA generated by the central build is the authoritative deployable artifact identity.

Use immutable tags for:

- automatic staging rollout
- incident analysis
- rollback targeting
- traceability between commit, image, and deployment

### Mutable tags

Mutable tags still exist:

- `dev`
- `stg`
- `prod` when manually published

Use mutable tags for:

- manual environment refresh
- human-readable environment semantics
- non-automated promotions

Do not rely on mutable tags as the authoritative record of what was actually deployed by an automatic workflow.

## Deployment use cases

### Case 1: change in a covered service

Expected outcome:

- image is published to GHCR
- staging is updated automatically
- the deployment references the immutable image built from that change

### Case 2: change in documentation or non-deployable repositories

Expected outcome:

- no runtime image publication
- no automatic staging deployment

### Case 3: cluster service update while ai-engine remains external

Expected outcome:

- covered services still auto-deploy to staging
- `ai-engine` runtime target remains a routing concern controlled from backoffice/gateway runtime state

### Case 4: controlled production release

Expected outcome:

- operator triggers manual deployment
- release uses explicit environment targeting
- rollout is observed and validated before completion

## Rollback strategy

### Staging rollback

Preferred approach:

1. identify the previous known-good short-SHA image
2. redeploy through `platform-infra` with explicit image selection or environment refresh process
3. verify rollout and smoke-test outcome

### Production rollback

Preferred approach:

1. restore previous release artifact or previous approved mutable `prod` state
2. re-run controlled deployment
3. confirm runtime stability and external connectivity

## Operational constraints

- externalized `ai-engine` means not every end-to-end behavior in staging is driven only by Kubernetes manifests
- routing overrides in `bff-backoffice` and `api-gateway` can influence effective runtime topology
- deployment documentation must therefore distinguish between cluster deployment state and effective runtime routing state

## Recommended release criteria

Before promoting to production, validate:

- service CI is green
- central image build is green
- staging rollout succeeded
- smoke paths for the public gateway are healthy
- contract compatibility is preserved
- runtime routing overrides are known and intentional
